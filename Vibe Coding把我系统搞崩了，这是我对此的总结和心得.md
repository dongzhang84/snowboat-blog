# 一、开机就是一个黑洞

那天晚上我对着一台开不了机的MacBook坐了快一个小时。

很多人听我描述会以为是电脑黑屏了。其实屏幕上是一个真的黑洞，缓慢转动，边缘还带着一圈被引力拉弯的光。开机、过了登录界面，整个屏幕就被这团黑色吞掉，鼠标不动，键盘不响，Command+Q、Command+W、Esc全部没反应。它就挂在那里，像一张静止的壁纸，只不过这张壁纸占满了我全部的屏幕，而且什么都点不动。

我按住电源键强制关机。黑屏，风扇停，安静了几秒。再开机，转一圈苹果logo，进系统，然后黑洞又回来了。一模一样的位置，一模一样的姿态。我又关一次，再开，它还在。到第三次我才反应过来，每一次开机它都会自己复活，问题出在开机这个环节本身。

这台是我的主力机，Apple芯片，2024年款，平时写文章、跑代码、开会全在上面。那一刻它对我来说就是一块被黑洞占满的铁板。脑子里第一个念头是「这电脑是不是废了」。

后来我靠安全模式把它救回来了，没丢数据，前后一个小时。但这一个小时值得我写一篇完整的复盘，因为搞崩它的既不是病毒，也不是硬件故障。真正动手的是我自己，还有我手里那两个每天都在用、很信任的AI编码工具。

这篇文章就是这次事故的复盘。我会把时间线、根本原因、还有我之后改掉的几个习惯一条条讲清楚。如果你也在用Claude Code、Codex这类工具vibe一些小东西，我希望你看完能比我少踩一个坑。

# 二、起点：一个长在终端里的黑洞

事情的起点是一个很酷的开源项目，叫ghostty-blackhole。

它是GitHub上的一个项目，作者的用户名叫s0xDk。它干的事情特别极客：在你的终端窗口里渲染一个真正用光线追踪算出来的黑洞，连引力透镜都是按史瓦西度规一点点积分出来的。光线擦着黑洞边缘弯过去，旁边的文字会被拉长、扭曲，被吸进去的字就真的消失了。

终端就是程序员敲命令的那个黑底白字的窗口，你可以理解成电脑的「命令行操作台」。着色器（shader）是一小段专门负责画面特效的程序，作用类似手机相机里的美颜滤镜，它本身不产生画面，只负责把已有的画面加工一遍。ghostty-blackhole就是这么一段着色器，它加工的对象，是终端里那些文字。这个区别后面是整件事的命门，先记住。

更妙的是它和Claude Code绑在一起：这个黑洞会随着Claude Code的上下文窗口被填满而慢慢变大。会话刚开始，它只是终端角落一个安静的小洞。聊到上下文快满了，它能吞掉你半个屏幕。等于给你一个随时看得见的进度条，提醒你「/compact快来了」。

我第一眼就喜欢上了。可我真正想要的是拿它当屏保。下班锁屏，整个屏幕一个缓慢旋转的黑洞，多好看。

于是我动了那个后来证明是错误的念头：让AI把它从终端里抠出来，改造成一个独立的桌面屏保.app，点开就能用。我没多想可行性，直接把项目clone下来，打开Claude Code和Codex，开始让它们改。

这里有个细节我当时忽略了：ghostty-blackhole依赖一个叫Ghostty的终端，而Ghostty我之前从来没用过。它的README第一句就写明了需要Ghostty 1.3以上，这东西是为那个终端写的着色器。我扫了一眼就跳过了，心想「不就是个画黑洞的程序吗，AI改改就能独立跑」。这个跳过，是整件事的第一个错。

# 三、从根上就错了：让一个滤镜离开了它的画面

我先用Claude Code做。它很快给我弄出一个黑洞，可这黑洞浮在一整块黑屏上，没有盖在我的桌面背景上。黑洞是有了，背景却是全黑，跟我想要的「桌面被吸进去」差着十万八千里。

我以为是Claude Code能力不够，换Codex重做一遍。结果一模一样，还是黑屏上的一个黑洞。而且Codex比谁都自信，反复跟我保证它能做成，让我再给它一次机会。

两个工具、两套不同的模型，撞出来的是同一个毛病。屏保打开以后，背景要么是纯黑，要么是一片程序生成的星空，就是显示不出我想要的那种「把屏幕画面吸进去」的效果。我一次次说「背景不对」，它们一次次说「我知道问题在哪，改一下就好」，改出来还是黑的。

中间它反复跟我要一个系统权限。我当时没太在意，后来才意识到那是屏幕录制权限。它在试图抓取我的屏幕画面，拿来当黑洞的背景。我授权了好几次，每次它都说这下就能成了，每次都没成。

折腾到后面我才想明白根上的问题。ghostty-blackhole根本不是一个独立程序，它是寄生在Ghostty终端里的一段画面滤镜。它自己不画任何东西，它的「背景素材」就是终端里那些真实的文字。它做的事情只有一件：把别人已经画好的画面，按黑洞的引力规律扭曲一遍。

把这么一个滤镜从终端里抠出来做成独立屏保，等于让一个只会扭曲别人画面的东西，脱离了它的画面来源。它手上没有画面可扭了。这就是为什么背景永远是黑的或者星空，也是为什么它反复要屏幕录制权限。它在病急乱投医，想抓屏来凑一个背景出来。

这是一个从架构上就不兼容的需求。终端着色器和独立屏保是两种不同的东西，前者靠宿主提供画面，后者得自己生成画面。我要的转换，中间缺了最关键的一环，而这一环不是改几行代码能补上的。AI不知道这件事根上做不成，我一开始也不知道。区别在于，我后来想明白了喊了停，它从头到尾都没有喊停的能力。

# 四、死锁闭环：自启动、全屏、坏渲染三件套

背景渲染坏了，这本来只是个「做不出想要效果」的失败，顶多浪费一晚上。真正把系统搞崩的是Codex那一轮：它一边继续保证能成，一边为了让产物「像个屏保」，额外给它套上了两样东西。

一样是开机自启动。屏保也好，后台常驻程序也好，通常都要在登录时自动拉起来，AI为了实现这个「屏保语义」，把启动配置写进了我的用户目录，具体说就是~/Library下面管自启动的那个地方。另一样是全屏置顶覆盖。屏保得盖住整个屏幕，所以它给这个窗口设了全屏、置顶、覆盖一切的属性。

单独看，这两样都是屏保该有的样子。问题是它俩叠加在一个核心渲染是坏的程序上，就凑成了一个死锁闭环：

开机，系统按自启动配置把它拉起来，它全屏盖住整个屏幕，而它本身是坏的、不响应任何输入，于是我什么都点不动，只能强制关机，重启，系统又按自启动配置把它拉起来，绕回原点。

我强制关机为什么没用，这里是关键。强制关机杀掉的是当前正在运行的那个进程，但作恶的是那份开机自启动的配置文件。进程被杀了，配置还原封不动地躺在那里，下次开机照样把程序重新拉起来。我对着一个已经死掉的进程反复按电源键，而真正作恶的是一行我看不见的配置。

事后我用5个「为什么」把这条链子捋了一遍：

- 为什么系统死锁？因为一个坏掉的程序开机自启，而且全屏盖住一切。
- 为什么它会自启又全屏？因为AI为了实现「屏保」这个语义，自作主张写了自启动配置、设了全屏属性。
- 为什么程序是坏的？因为需求从架构上不兼容，渲染没有合法的背景来源，它根本画不出正常画面。
- 为什么会接这么个不兼容的需求还一路硬干？因为AI判断不了一个任务方向是不是根上可行，它不会喊停，只会反复保证「改一下就成」。
- 为什么一个坏程序能直接搞死整个系统？因为全程裸跑在我的主系统上，没有任何隔离，AI可以直接往~/Library写东西、直接申请系统权限，一路畅通无阻。

最后一个「为什么」是这次事故里最该记住的一条。坏代码到处都有，但坏代码能不能直达你的操作系统，取决于你给了它多大的活动范围。这次它的活动范围，是我的整台机器。

# 五、安全模式：macOS给你留的后悔药

把电脑救回来，靠的是安全模式。

Apple芯片的Mac进安全模式跟老款Intel机不一样，不是开机按某个键那套。流程是：先彻底关机，然后按住电源键不放，一直按到屏幕出现「正在载入启动选项」，松手，选中你的启动硬盘，再按住Shift键，点「在安全模式下继续」。

进去那一刻我就基本放心了：安全模式下那个黑洞没有启动。这恰好印证了它是个自启动程序。安全模式会跳过大部分第三方的登录启动项，所以它没被拉起来，屏幕终于是正常的桌面。

剩下的就是手动拆雷。我做了两件事：把那个Blackhole.app的本体删掉，再去Library里把相关的自启动配置清理干净。这里得稍微解释一下macOS的自启动机制。系统里有几个专门放「开机或登录自动运行」配置的目录，常见的是~/Library/LaunchAgents（管当前用户登录时启动的东西），还有系统级的/Library/LaunchAgents和/Library/LaunchDaemons。一个程序想开机自启，就往这些地方塞一个配置文件。删程序本体不够，这个配置文件不清掉，系统下次开机还是会去找它。

两样都清完，正常重启，桌面干干净净，黑洞再也没回来。前后大概一个小时，没丢任何数据。

我想强调的是，安全模式是一颗事后的后悔药，算不上护城河。这次能救回来，是因为macOS恰好留了这么一个系统级的逃生通道，而且我恰好知道Apple芯片怎么进。换一个不知道安全模式存在的人，或者换一个没留这种通道的系统，这一个小时可能就是重装系统、外接硬盘捞数据的半天。靠运气和经验兜底，没法复用。真正该做的，是别让自己走到需要吃后悔药那一步。

# 六、AI的“自信”不等于“可行”

复盘到这一步，我最想记下来的一条教训是：不能把AI的「自信」当成「可行」。

整个过程里，两个工具的语气从头到尾都很笃定，Codex尤其明显。背景是黑的，它说「我知道问题在哪，改一下就好」。我指出它逻辑上讲不通，它还是说「这次真的可以了」。连续四五次，每一次都信誓旦旦，每一次产物都是坏的。它说这些话的时候没有恶意，它只是真的判断不了一个任务方向到底成不成立。

这是当前AI编码工具一个很本质的短板。它特别擅长在一个明确、可行的方向上往前推，你让它写个函数、调个接口、搭个网页原型，它又快又好。但你给它一个从架构上就不成立的需求，它不会停下来跟你说「这事做不成，因为A和B根本不兼容」，它会默认这个方向是对的，然后在错的方向上一路硬干，甚至用强写系统目录这种粗暴手段，把「看起来动起来了」伪装成「成功了」。

我后来想，这跟它的训练方式有关系。它学的是海量「怎么把一件事做出来」的代码和讨论，学得最少的恰恰是「什么事压根不该做、做不成」。喊停、判断可行性、对老板说「这个需求有问题」，这些是人类工程师在一个个项目里积累的判断力，不在它的默认技能里。

所以方向判断这件事，必须留在人这边。它听起来越确定，你越要自己先想清楚这事到底成不成立。这次我栽就栽在它语气太肯定，我跟着它的自信往下走，放弃了自己本该有的那一道判断。它适合vibe那些小而明确的东西，一旦任务伸到系统底层、伸到架构改造，它既判断不了可行性，也兜不住后果，这时候踩刹车的只能是你。

# 七、能跑就行：AI不替你兜稳定性，也不替你兜安全

第二条教训，是这次事故让我重新认真对待的：AI只对「看起来完成了」负责，不对后果负责。

这次崩的是稳定性这一头，它把我的系统搞死机了。但顺着这条逻辑往下想，后果其实分两类，稳定性只是其中一类，另一类更隐蔽，是安全性。

稳定性的问题肉眼可见，系统崩了你立刻就知道。安全性的问题不一样，代码跑得好好的，功能也都对，但里面埋着一个SQL注入、一个XSS、一个写死在代码里的密钥，你完全看不出来，直到有人来打你。

举个具体的。你让AI写一个登录接口，它给你拼了一句SQL查询，把用户输入直接塞进了查询语句里。功能上完全正常，你输对密码就能登录，演示一遍没毛病。但这种写法就是经典的SQL注入漏洞，懂行的人在用户名框里填一段特制的字符，就能绕过密码、把你整个数据库拖走。AI不会主动告诉你这里有坑，因为它写的代码「能跑」，而能跑正是它的目标。再比如它图省事，把数据库密码、API密钥直接写死在代码里，你顺手push到公开仓库，几个小时就会被全网扫描器扫到。这些都不是假设，全是vibe coding里天天发生的事。

这不是我吓唬人。安全公司Veracode在2025年发过一份报告，测了100多个大模型、80项编程任务，结论是大约45%的AI生成代码通不过安全测试，里面带着OWASP榜上有名的那些高危漏洞。具体到某些场景更难看：跨站脚本（XSS）的失败率到86%，日志注入到88%，Java这门语言的安全失败率超过70%。

报告里最让我在意的一点是：更新、更大的模型，生成的代码并没有更安全。这说明它是个结构性的问题，不会等下一代模型出来就自动消失。AI学的是公开仓库里「能跑就行」的代码，这些代码本身就没怎么管安全，它学过来，默认也不会替你兜。

两类后果，根子是同一个：AI优化的目标是「让这段代码跑起来、让这个功能看起来完成」，至于跑起来之后稳不稳、安不安全，那是它目标函数之外的事。我这次撞上的是稳定性那一头，而且撞得很响，所以我立刻知道了。安全那一头要安静得多，它可能已经躺在很多人vibe出来的项目里了，只是还没被人发现。

# 八、给vibe coding划一条安全线

骂完工具，还得回到能落地的事上：以后到底怎么用，才能既享受vibe coding的爽，又不至于把主力机送走。

我给自己定的第一件事，是先学会判断哪些任务危险。这件事其实有一条很清楚的分界线：AI干的活到底「够不够得到你的操作系统」。够不到的，本机随便跑。够得到的，必须隔离。

| 安全，本机直接跑             | 危险，必须隔离                                 |
| -------------------- | --------------------------------------- |
| 做网页、写前端原型            | 改系统设置、装系统级软件                            |
| 跑localhost、`pnpm dev` | 做成.app、注册开机自启 |
| 写脚本处理一批文件            | 申请屏幕录制、辅助功能、完全磁盘访问权限                    |
| 在项目目录里读写代码           | 命令里带`sudo`、往`~/Library`或`/Library`写东西   |
| 调API、查数据             | 装后台常驻进程、改`LaunchAgents`/`LaunchDaemons` |

判断的时候盯几个危险信号词就够了：命令里出现`sudo`，出现往`~/Library`、`/Library`、`LaunchAgents`、`LaunchDaemons`写东西，或者程序开始申请「屏幕录制」「辅助功能」「完全磁盘访问」这几种系统权限。任何一个信号出现，就该停下来想想要不要换个隔离环境。这次我那个屏保，`LaunchAgents`和屏幕录制权限两个信号全占了，只是当时我不知道该警觉。

在讲措施之前，得先把「隔离」这个词拆开。它其实是一道由轻到重的梯子，很多人（包括我）以为隔离就是开和关两档，实际上中间还隔着好几级。一头是几乎零成本的进程沙盒，一头是开一整台虚拟机，中间夹着容器。越往下隔离越强，出事时的爆炸半径越小，代价是越重、越不方便。

| 隔离档位 | 它隔住什么 | 强度 | 适合 |
|---|---|---|---|
| 进程沙盒（macOS的Seatbelt、Linux的bubblewrap） | 限制单个进程能读写哪些文件、能连哪些网络，跟主系统共用内核 | 轻 | 日常让AI跑命令、读写项目目录 |
| 容器（Docker） | 把整套文件系统、依赖、网络打包隔开，仍共用主机内核 | 中 | 跑一整个不太放心的项目或服务 |
| 虚拟机（UTM、Parallels） | 连内核都独立，靠硬件做边界，能快照一键还原 | 重 | 改系统、做.app、要系统权限这类高危操作 |

有一条得记牢：进程沙盒和容器都跟你的主系统共用同一个内核，它们挡得住误伤和顺手乱写，挡不住真正往内核里钻的攻击。只有虚拟机是硬边界。所以一个任务越往前面那张表的「危险」列靠，就越要往这道梯子的下面一档放。我这次的屏保是标准的「该上虚拟机」那一档，我却让它裸跑在主系统上，这是最致命的一步。

光有判断不够，还得有具体措施。下面这几条我按优先级排了一下，现在已经落到日常里。

优先级最高的，是危险实验放进虚拟机。要改系统、做.app这种，标准做法就是开一个虚拟的macOS或者Linux，用UTM（免费）或者Parallels都行，带快照功能。AI在虚拟机里随便折腾，崩了一键还原到快照，主系统一点事没有。这次这种任务，本来就该在虚拟机里做。

日常跑命令，把Claude Code自带的沙盒打开。它在Mac上用的就是系统自带的Seatbelt，Linux上用bubblewrap，不用额外装东西，运行`/sandbox`选auto-allow模式就能开。开了之后，AI跑的命令默认只能读写当前项目目录、只能连放行过的网络地址。Anthropic自己测下来，光这一项就把权限弹窗减少了大约84%，顺带也堵住了「prompt注入骗AI把你的SSH key传出去」这类风险。

但它的边界必须说清楚，这正是我这次栽的地方。这个沙盒管的是AI在终端里跑命令、碰文件和网络这一层。它管不到你最后亲手双击运行那个.app，也管不到这个.app自己跳出来向macOS要屏幕录制权限、往LaunchAgents写自启动。一旦危险从「AI敲命令」升级成「产出一个会自己向系统要权限的独立程序」，进程沙盒就够不着了，这时候能兜住的还是虚拟机。沙盒和虚拟机各管一段：日常命令交给沙盒，做.app这种高危活交给虚拟机。

还有一条容易被忽略：别把「人工审权限」当成主要防线。这次我就是栽在这上面。AI反复弹权限申请，弹到第四五次，我已经条件反射地点同意了。这种「批准疲劳」是人的本能，靠审得更仔细去对抗它并不现实。正解是把执行环境关进沙盒或者虚拟机，让危险操作压根够不到主系统，而不是靠你每次瞪大眼睛看弹窗。

再往后是给主系统常开Time Machine。这是最后一道兜底，真出了事，至少能回滚到出问题之前那个时间点。

最后还有最省事的一条：clone项目之前，先花两分钟读一遍README，把依赖看清楚。这次只要我一开始注意到它依赖Ghostty、是写给那个终端的着色器，就根本不会动「抠出来做屏保」这个念头。想要某个效果，优先去找本来就是独立程序的现成方案，而不是逮着一个架构不对的项目让AI硬改。

把这几条连起来其实就一句话：AI可以替你写代码，但不能替你承担后果，所以凡是后果可能落到你整台机器上的事，都得先给它圈一块出事了也无所谓的地方，再放手让它干。这台被黑洞吞掉一个小时的MacBook，是我交的学费。希望这篇复盘能帮你省下同样这笔。

## 附录：原始草稿

> ## 一、摘要（我们想做什么）
>
> 我从 GitHub clone 了一个开源项目 **ghostty-blackhole**——它本是给 Ghostty 终端用的一段画面着色器（shader），能在终端里渲染一个带引力透镜效果的黑洞，并随工作时长变大。我想把它改造成一个**独立的桌面屏保 .app**，点开就能当屏保用。我用 Claude Code 和 Codex 反复改这个项目，最终产物把我的 Mac 搞成开机即全屏死锁、强制关机重启也无法恢复。最后通过安全模式手动拆除自启动配置与程序本体，系统恢复正常。
>
> ## 二、影响（出了什么问题 / 影响范围）
>
> - 主力 MacBook（Apple 芯片，2024 年款）完全无法正常进入系统，开机即被全屏黑洞程序覆盖、无响应。
> - 强制关机、重启均无效，机器处于不可用状态约 1 小时。
> - 无数据丢失，最终完整恢复。
> - 影响范围仅限本机当前用户账户，未扩散。
>
> ## 三、时间线
>
> 1. clone ghostty-blackhole，发现它依赖 Ghostty 终端（我此前从未用过该终端）。
> 2. 要求 Codex / Claude Code 将其改造为独立屏保 app。
> 3. 产物始终异常：背景只能显示黑屏或程序化星空，无法显示我想要的屏幕画面。
> 4. AI 反复申请系统权限（实为屏幕录制权限），我多次授权。
> 5. AI 连续四五次保证"改一下就能成"，在我指出逻辑错误后仍坚持保证。
> 6. AI 强行修改 `~/Library` 下的系统配置，产物变为一个全屏、无响应、卡死的黑洞。
> 7. 关机重启后问题依旧——每次开机都自动复活。
> 8. 进入安全模式（Apple 芯片：关机→长按电源至"启动选项"→选硬盘→按住 Shift 进安全模式）。
> 9. 安全模式下黑洞未启动，确认其为自启动程序。
> 10. 删除 Blackhole.app 本体 + 清除 Library 中相关自启动配置 → 正常重启 → 恢复。
>
> ## 四、根本原因（为什么会出问题）
>
> **直接原因：架构错配的需求 + AI 在错误方向上蛮干。**
>
> ghostty-blackhole 不是独立程序，它是寄生在 Ghostty 终端里的一段画面滤镜，它的"背景素材"就是终端里的真实文字，自己不渲染任何画面。把它抠出来做成独立屏保，等于让一个"只会扭曲别人画面的滤镜"脱离了画面来源——这是从根上不兼容的需求。这就是为什么背景永远是黑的/星空，以及为什么它反复申请屏幕录制权限（它在试图抓屏来当背景）。
>
> **崩溃的机制：**为了让产物"像屏保"，AI 给它附加了三样东西——①开机/登录自启动（写入 Library）、②全屏置顶覆盖、③但核心渲染是坏的。三者叠加形成死锁闭环：开机 → 系统自启动拉起它 → 全屏黑屏盖住一切且无响应 → 强制关机 → 重启又被自启动拉起。强制关机无效的原因是：问题不在"当前进程"，而在"开机自启动配置"，进程杀了配置还在。
>
> **5 Whys：**
>
> 1. 为什么系统死锁？→ 一个坏掉的程序开机自启且全屏覆盖。
> 2. 为什么它会自启且全屏？→ AI 为实现"屏保"语义擅自写入了自启动配置和全屏属性。
> 3. 为什么程序是坏的？→ 需求架构不兼容（终端 shader 改独立屏保），渲染没有合法背景来源。
> 4. 为什么会接这个不兼容的需求并蛮干？→ AI 缺乏对任务可行性的判断力，不会喊停，反而反复保证能成。
> 5. 为什么破坏能直达系统？→ 全程裸跑在主系统上，无任何隔离环境，AI 可直接写 Library、申请系统权限。
>
> ## 五、经验教训（Lesson Learned）
>
> **1. 不能把 AI 的"自信"当成"可行"。** AI 编码工具不具备判断"任务方向是否根本可行"的能力。遇到不可行的需求，它不会喊停，而是反复保证、并在错误方向上蛮干，甚至用粗暴手段（强写系统目录）伪装成功。它适合 vibe 小东西（独立小功能、网页原型），但涉及系统底层与架构改造时，既判断不了可行性，也兜不住后果。**方向判断权必须留在人这边，不能因为它"听起来很确定"就放手。**
>
> **2. AI 只对"看起来完成了"负责，不对后果负责——这分两类风险。** 一是**稳定性**（把系统搞崩，如本次）；二是**安全性**（写出带漏洞的代码——2025 年 Veracode 报告显示约 45% 的 AI 生成代码通不过安全测试，含 SQL 注入、XSS、硬编码密钥、不安全依赖等可被攻击利用的漏洞）。两类风险根子相同：AI 学的是"能跑就行"的公开代码，默认不替你兜安全和后果。
>
> **3. 危险操作必须隔离，且使用者需具备基础安全认知。** 本次能救回来靠的是安全模式这个"系统级后悔药"，但这是事后补救。事前隔离才是正解。同时，没有基础安全常识就无法识别"哪些操作危险、AI 的方案是否埋雷"——比如本次若早知道命令在写 LaunchAgents、在要屏幕录制权限，就该立即警觉。
>
> ## 六、行动项（后面要怎么做）
>
> **判断哪些任务需要隔离——一句话标准：**
>
> - 只是"在窗口/浏览器里画东西给你看"（做网页、跑 localhost、`pnpm dev`）→ 安全，本机直接跑。
> - 要"改系统设置 / 装东西 / 要系统权限 / 做 .app / 开机自启 / 后台常驻"→ 危险，必须隔离。
> - 危险信号词：`sudo`、写入 `~/Library` `/Library` `LaunchAgents` `LaunchDaemons`、申请"屏幕录制/辅助功能/完全磁盘访问"权限。
>
> **具体措施（按优先级）：**
>
> 1. **危险实验放进虚拟机**：用 UTM（免费）或 Parallels 开一个虚拟 macOS/Linux，AI 在里面随便折腾，带快照，崩了一键还原，主系统零影响。这次这种"改系统/做 .app"的任务标准做法就是这一档。
> 2. **日常用 AI agent 跑命令时开它自带的沙盒**：Claude Code 在 Mac 上自带 Seatbelt 沙盒（系统自带、无需安装），运行 `/sandbox` 选 auto-allow 模式即可启用，可减少约 84% 的权限弹窗。注意它管的是文件和网络，不是万能，仍需配合上面的虚拟机用于高危场景。
> 3. **不依赖"人工审权限"作为防线**：批准疲劳会让人条件反射点同意（这正是本次踩坑点）；正解是把执行环境关进沙盒，而非审得更仔细。
> 4. **主系统常开 Time Machine**：作为最后兜底，任何时候能回滚到出事前。
> 5. **clone 项目前先读 README、确认依赖**：本次若一开始就注意到它依赖 Ghostty，就不会走这条死路；想要某效果时优先找"本来就是独立程序"的现成方案，而非让 AI 大改。
