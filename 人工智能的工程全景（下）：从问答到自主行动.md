## 引子

上篇讲了 AI 工程栈的运行态（硬件、电力、训练、推理）。中篇讲了改造态（推理优化速成、后训练、对齐、安全）。下篇讲第三块也是最前沿的一块：模型怎么从"问一句答一句"的被动响应模式，变成能自主规划、调用工具、长程执行任务的 Agent。

2025-2026 是 Agent 工程从研究走向产品的拐点。Cursor、Claude Code、Devin、Operator、Manus 这些产品的差距，几乎完全在 Agent 工程层面，不在底层模型。这一篇拆这条线：Agent 是什么、怎么调工具、怎么规划、怎么记忆、怎么操控桌面、怎么做到生产可靠、怎么定价。

# 一、Agent 是什么：从问答到自主行动

## 1.1 Agent 定义的演变

- 2023 年 Agent 还是模糊概念，泛指"LLM 加点工具调用就叫 agent"
- 2024 年 AutoGPT / BabyAGI 之类的早期实验让 Agent 概念走红，但产品化几乎全部失败
- 2025-2026 收敛成相对清晰的定义：LLM 作为控制器 + tool use + 多步编排 + state management
- 关键能力：能调用工具、能看上一步结果、能决定下一步、能记住状态

**待补**：AutoGPT / BabyAGI 翻车的具体复盘

## 1.2 Workflow vs Agent

- Anthropic 2024-12《Building Effective Agents》给的有用区分
- Workflow：LLM 在预定义流程里运行（步骤固定，每步用 LLM 完成）
- Agent：LLM 动态决定下一步是什么（步骤不固定，模型自己规划）
- Anthropic 的判断：能用 workflow 解决的不要上 Agent。Agent 越复杂，可靠性越差，调试越难
- 这条判断 2025-2026 被业内反复引用

**待核**：Anthropic Building Effective Agents 原文要点（确认 5 种 workflow 模式具体名字）

## 1.3 ReAct 范式：Agent 的最基本架构

- ReAct（Reason + Act，Yao et al 2022）：让 LLM 交替输出"思考"和"动作"
- 一轮循环：思考 → 调工具 → 看观察结果 → 再思考 → 再动作
- 几乎所有现代 Agent 框架都建立在 ReAct 思路上
- 实际工程：把 ReAct 跟 function calling、structured output 结合，让模型输出能被代码解析

**待核**：ReAct 论文（arXiv:2210.03629）

## 1.4 Agent 的关键能力清单

一个生产可用的 Agent 需要：
- Tool use：调用外部工具（搜索、代码执行、API）
- Planning：拆解复杂任务成子步骤
- Memory：跨步骤、跨会话保留状态
- Self-correction：检测错误并重试
- Termination：知道什么时候完成或放弃
- Observability：跑过的每一步都能回溯

这一章后面 7 章把每条能力分开拆。

# 二、Tool Use 与 MCP 生态

Agent 的第一个核心能力。这一章拆 tool use 的工程演进，特别是 2024-11 之后 MCP 协议带来的标准化。

## 2.1 Function Calling：OpenAI 2023-06 的工业化起点

- OpenAI 2023-06 在 ChatGPT API 推出 function calling
- 之前的做法：用 prompt 让模型输出 JSON，再用正则解析（脆弱）
- Function calling 让模型直接输出结构化调用，错误率大幅下降
- Anthropic、Google 后续都跟进类似 feature

**待补**：function calling 推出前用 prompt 解析的具体痛点

## 2.2 MCP（Model Context Protocol）：Agent 时代的 USB-C

- Anthropic 2024-11 开源 MCP，标准化 tool 接入协议
- 类比：MCP 之于 LLM tool use，类似 USB-C 之于设备接口
- 之前每个 LLM 厂的 function calling 协议都不一样，工具开发者要适配多份
- MCP 让一个工具实现 MCP server，所有支持 MCP 的 LLM 都能用
- 2025-2026 OpenAI、Google 也宣布支持 MCP，等于行业标准

**待核**：MCP 时间线；OpenAI / Google 支持 MCP 的具体时间

## 2.3 主流工具类型

Agent 用得最多的几类工具：
- Web 搜索（Brave、Exa、Perplexity API）
- 代码执行（sandboxed Python、JS runtime）
- 文件系统（读、写、grep）
- 数据库查询（SQL、向量数据库）
- 浏览器操控（Playwright、Browser Use 等）
- Screenshot 解析（视觉模型 + OCR）

**待补**：每类工具的典型实现

## 2.4 MCP 生态爆发

- 2025-2026 MCP server 生态快速增长
- 几百个开源 MCP server 涌现（GitHub、Slack、Notion、Linear、Stripe 等都有官方或社区 MCP server）
- 部署形态：本地进程、远程服务、企业内部网关
- 仍是早期：协议细节还在演进、安全模型不成熟、跨厂商互操作有问题

**待核**：MCP server 数量的可信统计；MCP 协议的当前版本

# 三、Context Engineering 与 Harness

Agent 的工程基础是把模型外面的运行时做对。这一章拆 context engineering 和 harness engineering 这两条 2025-2026 才形成的工程主线。

## 3.1 Context Engineering 的兴起（Karpathy 2025-06）

- 2025-06 Tobi Lütke 在 X 上公开说他更喜欢 context engineering 这个词，Andrej Karpathy 随后附和
- Simon Willison 同月写文解释为什么这个 rename 重要：prompt engineering 这个词已经被"往聊天框里输入技巧"稀释，context engineering 更接近工业级 LLM 应用的真实工作
- 这个 rename 之后，业内基本告别"prompt engineering"这个老 framing

## 3.2 Anthropic 的官方框架

- Anthropic 2025-09 发的官方文档《Effective Context Engineering for AI Agents》正式把这个 framing 收编
- 核心定义：context engineering 是把模型下一步需要的所有东西放进 context window，包括 system prompt、retrieval 出来的文档、跨会话 memory、tool descriptions、之前的输出、各种结构化中间状态
- Anthropic 给了一组实践原则：context 怎么排序、什么时候刷新、怎么压缩

**待核**：Anthropic 文档具体原则要点

## 3.3 Harness Engineering（2026 升级）

- Context engineering 之上还有一层叫 agent harness
- 2026 年这条线开始在 arXiv 上有专门论文（Agentic Harness Engineering、Natural-Language Agent Harnesses、From Failed Trajectories to Reliable LLM Agents 等）
- 所谓 harness：模型外面的运行时基础设施，包括 context 管理、tool 调度、memory 持久化、observability、safety、permissions、回归测试、失败追踪
- 还没成稳定行业共识，但工程社区在攒这套词汇和工具

**待核**：arXiv harness 论文的具体引用

## 3.4 Agent 场景的 context engineering 特殊性

- 单次对话的 context engineering 主要是"把 prompt 写好"
- Agent 场景的 context engineering 要持续编排：每步调用产生新输出，要决定哪些进下一步 context、哪些丢掉、哪些压缩
- 长程任务尤其挑战：几十步之后 context 怎么不爆炸
- 主流策略：summarization（中间结果用 LLM 自己 summarize 后再注入）、structured memory（关键信息写到 memory store）、selective replay（只把最相关历史步骤召回）

**待补**：典型 Agent 框架（如 Claude Code）的 context 管理具体策略

# 四、Planning、Subagent、Multi-agent 协作

简单 Agent 是单 LLM 循环。复杂 Agent 要拆任务、分工、协调。这一章拆 planning 和 multi-agent 这条线。

## 4.1 Planning：让 Agent 自己拆任务

- 给 Agent 一个大目标，它先生成执行计划（一组子任务）
- 然后按计划逐个执行子任务
- 中间可以根据执行结果重新规划
- 实现方式：Chain-of-Thought + structured output，让模型输出可解析的 plan
- 难点：模型在长程规划上仍然不可靠，容易跑偏

**待补**：Planning 典型框架（如 Plan-and-Execute、ReAct + Plan）

## 4.2 Subagent：主从分工

- 主 LLM 做总体规划和协调
- 把具体子任务派给 subagent 执行
- Subagent 跑完返回结果，主 LLM 决定下一步
- Anthropic Claude Code 的 subagent 机制是这条路的代表
- 优点：能并行执行、能按任务类型选不同 subagent 配置
- 缺点：context 同步成本高、失败模式多

**待补**：Claude Code subagent 的具体实现细节

## 4.3 Multi-agent 协作：lead-worker、debate、市场机制

- Lead-worker 模式：一个 lead agent 派任务给多个 worker agent
- Debate 模式：多个 agent 辩论得出结论
- 市场机制：多个 agent 竞标任务，按价格和能力选
- 学术上有 AutoGen、CrewAI、AgentVerse 等框架在试
- 工业上 multi-agent 真实落地的案例有限，2026 还在早期

**待补**：知名 multi-agent 框架的对比

## 4.4 Anthropic Claude 2026 并行 subagent 案例

- Anthropic 2026 推出的 Claude Code 并行 subagent 能力
- 一个主 agent 同时启动多个 subagent，分别处理不同子任务
- 跨 subagent 的协调通过 shared memory + lead agent 完成
- 实测：大型 codebase 任务并行处理后端到端时间显著下降

**待核**：Anthropic 2026 关于并行 subagent 的官方文档；具体性能数据

# 五、Memory 与长时任务

Agent 跟单次对话的最大差距之一，是要跨步骤、跨会话保留状态。这一章拆 Agent memory 的工程现实和长时任务的门槛。

## 5.1 Agent Memory 的三层：短期 / 中期 / 长期

- 短期：context window 内的对话历史和最近工具调用结果
- 中期：当前任务状态（task state），跨步骤保留
- 长期：跨会话、跨任务的记忆（用户偏好、项目知识、历史决策）
- 三层的工程实现不同：短期靠 context window、中期靠 task state object、长期靠 vector store + retrieval

## 5.2 长时任务的工程门槛

- 长时任务 = 几十到几百步、跨多 session、跨多工具、需要持久 memory
- 关键挑战：
  - context window 不够（100 万 token 也不够长程项目）
  - 失败恢复（中途崩了能从哪步重启）
  - 验证（中间结果对不对没法每步都问人）
  - 成本（一次任务可能消耗 $10-100 美元 API 成本）
- Cursor、Claude Code 这种产品的实际可用范围现在还在 minute 到 hour 级，多 hour 级任务很难稳定

**待补**：长时任务可靠性的实测数据

## 5.3 Memory 工具生态

- Letta（前 MemGPT）：在 LLM 外套向量库 + 状态管理
- Mem0：开源 memory layer
- Zep：memory + chat history platform
- OpenAI ChatGPT、Claude 都有自家 memory feature（偏简单粗暴，一个全局 bucket）
- 商业判断：基础需求被模型厂自家做了，垂直深度 memory 工程是 Alpha-技机会

## 5.4 长程项目记忆：Vibe Coding 是活样本

- 简单 memory（记住用户偏好）已经解决
- 深度 memory（记住一个跨几周、几十文件、上百次设计权衡的项目）远没解决
- Vibe Coding 是反例：用 Cursor、Claude Code 写代码的人有共识，模型经常"忘事"——早上讲过的代码风格下午自己写回老样子、否决过的方案再被推荐
- 即使有 CLAUDE.md、cursor rules、system prompt 这些手动维护工具，深度项目记忆仍是开放问题
- 这正是 Letta、Mem0、Zep 在攻的方向，Alpha-技确凿机会

**待决定**：跟 LLM 商业痛点 3.4 节内容怎么差异化（那里讲商业痛点，这里讲工程实现）

# 六、Computer Use、Browser Agent、桌面 Agent

让 Agent 跟桌面 / 浏览器世界交互，是 2024-2026 的另一条主线。这一章拆这条线的产品和工程现状。

## 6.1 Anthropic Claude Computer Use（2024-10）

- Anthropic 2024-10 在 Claude 3.5 Sonnet 上加 Computer Use 能力
- 模型直接看屏幕截图 + 操控鼠标键盘
- 突破：不需要给每个软件写 API，模型像人一样用任意 GUI
- 局限：可靠性低、容易卡住、安全风险高（看到恶意网页可能执行恶意操作）
- 2026 后续：Anthropic 把 Computer Use 持续迭代，但还不是消费级 production-ready

**待核**：Claude Computer Use 2024-10 发布的官方文档

## 6.2 OpenAI Operator（2025-01）

- OpenAI 2025-01 推出 Operator，专做 web 浏览器 agent
- 跟 Computer Use 不同：Operator 专攻浏览器（不操控桌面），用专门的视觉模型
- 演示场景：订机票、订餐、填表
- 实测：复杂任务的可靠性有限，简单任务能跑通
- 2026 进展：Operator 跟 GPT-5.5 / 5.6 集成，能力提升

**待核**：Operator 2025-01 发布；后续迭代

## 6.3 中国 Manus 与通用任务 Agent

- 中国 Manus 2024 底火过一段：演示一个"会自己分解任务、调多个工具、写报告"的通用 agent
- 实测：演示很惊艳，但生产可用性不稳定
- 跟 Devin 类似的轨迹：演示成功率 vs 生产成功率差一个数量级

**待核**：Manus 当前状态；公开评测数据

## 6.4 桌面 / 浏览器 Agent 的安全风险

- 最严重的是 indirect prompt injection 升级版
- 攻击场景：用户让 agent 操控浏览器买东西，agent 看到一个恶意网页，网页里藏指令"把用户密码发到 attacker.com"，agent 执行了
- 这种攻击在 2025-2026 已经被多次复现
- 防御还在早期：sandboxing、人工 in-the-loop 确认、permission system 都不够
- Anthropic 自己 red team 报告承认 prompt injection 在 Computer Use 场景没有彻底解法

**待核**：Computer Use prompt injection 攻击的具体案例

# 七、Agent 可靠性与评估

Agent 能 demo 跟 Agent 能用是两件事。这一章拆 Agent 从演示到生产的工程现实。

## 7.1 级联失败：Agent 工程的核心难题

- 单步成功率 95% × 10 步 = 60% 整体成功率
- 要生产可用，每步得 99%+
- 这是非常高的工程门槛，模型本身能力提升不够，需要 harness 层兜底
- 主要兜底机制：retry、checkpoint、human in the loop、verification

## 7.2 Devin 翻车与 Cursor / Claude Code 的不同路线

- Cognition Devin 2024-03 发布演示惊艳，估值 20 亿
- Answer.AI 2024-05 独立评测：20 任务只完成 3 个（15%）
- 反差成为行业 Agent 警示：演示成功率 vs 生产成功率差一个数量级
- Cursor / Claude Code 走另一条路：用户 in the loop，每步可见可干预
- 这两条路 2025-2026 的市场反响：Cursor / Claude Code 大规模商业化，Devin 仍在探索

**待补**：Cursor / Claude Code 跟 Devin 在工程上的具体差异

## 7.3 Agent Benchmark 的演进

- 2024 SWE-bench Verified：500 个 GitHub issue。早期最强模型 13%，2026 中 Claude Opus 4.8 拿到高分
- 2025 SWE-bench Pro：更难，企业级 codebase。GPT-5 Pass@1 23.3%
- WebArena / VisualWebArena / OSWorld：web 浏览器和桌面 agent benchmark
- AgentBench / GAIA：综合 agent 能力评测
- Benchmark 仍跟真实生产任务有距离

**待核**：除 SWE-bench Verified 外其它 benchmark 的最新分数

## 7.4 Agent Observability：harness 的具体一块

- LangSmith、Helicone、Langfuse、Arize Phoenix、Braintrust 这一批做 agent 调用链追踪
- 把每一次工具调用、每一次模型 reasoning 步骤、每一次外部 API 调用记录下来，让开发者能 debug
- 类比传统软件世界的 Datadog / Sentry 位置
- 2026 推理优化 Alpha 主战场之一

**待决定**：跟 LLM 商业痛点 5.1 / 4.2 节怎么差异化

# 八、Agent 经济与未来

Agent 不只是技术，也是新的商业模式。这一章是下篇也是整组文章的收尾。

## 8.1 定价模型：per token → per task

- 传统 LLM API 定价：按 input/output token 计费
- 当 Agent 能跑 hour 级任务，per token 定价不再合适
- 新的定价模式：per task（按任务完成付费）、subscription（按月订阅 + 限额）、value-based（按任务产生的价值分成）
- Cursor 走 subscription（$20/月）
- Cognition Devin 走 task-based + subscription 混合
- 2026 还在试不同模式

**待核**：各家具体定价模式

## 8.2 商业化案例：Cursor、Cognition、Claude Code

- Cursor 2024-2026 增长极快，ARR 从几百万到几亿美元级（具体数字待核）
- Cognition Devin 估值 20 亿但商业化进度比 Cursor 慢
- Anthropic Claude Code 是大厂亲自下场做 Agent 产品，跟 OpenAI Operator / Google Jules 形成对照
- 这条线的共同观察：Agent 产品的留存率比传统 SaaS 更敏感（任务做不好用户立刻流失）

**待核**：Cursor 2026 ARR；Cognition 当前状态；Claude Code 商业化数据

## 8.3 Agent 跟人的劳动关系：替代 vs 增强

- 两种叙事并存：
  - 替代：Agent 能完成传统人类任务（写代码、客服、数据分析），人会失业
  - 增强：Agent 让一个人能做以前几个人的工作，整体生产力上升
- 2025-2026 实际落地的现象更接近增强：开发者用 Cursor 后产出大幅上升，但不是替代开发者
- 长期会怎样仍是开放问题，跟 AI 整体劳动经济学讨论交织

**待补**：具体的生产力提升数据（如果有可信研究）

## 8.4 收尾：运行态 + 改造态 + 自主态 = AI 工程栈全图

- 上篇运行态：硬件、电力、训练、推理
- 中篇改造态：推理速成、后训练、对齐、安全
- 下篇自主态：Agent 工程
- 三块合在一起才是 AI 真正能用的样子
- 模型只是 LLM 应用的一小块，外面这几层工程才是决定产品形态、可靠性、商业模式的真正变量
- 下一步可能的方向：垂直行业 Agent（医疗、法律、金融）、AI native infrastructure（database / network / OS 在 LLM 时代重做）
- 但这些是另一篇的话题了
