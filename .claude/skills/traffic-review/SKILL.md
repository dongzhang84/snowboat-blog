# 流量复盘

对一段时间内发表的文章做流量归因分析，产出一份 analysis/*.md。

## 第零步：加载配置（必做）

开始分析前，用 Read 依次加载：

1. `.claude/skills/traffic-review/framework.md`：分析框架（高流量四支柱、低流量失败模式、评分口径）
2. `.claude/skills/traffic-review/output-template.md`：输出文件结构

把这两份内容作为硬性框架。

## 第一步：确定范围

- 从 `$ARGUMENTS` 解析时间范围或文章编号
- 如果 `$ARGUMENTS` 为空，默认取 README 中最近 5-7 篇有流量数据的文章（跳过 `—`）
- 不接受未来日期

## 第一·五步：必须先读 pending-notes（如果存在）

- `ls analysis/pending-notes-*.md` 查找是否有临时观察文件
- **如果有，必须 Read 一遍**。pending-notes 里常常记着只有当时才能观察到的关键事实：首小时实时数据、限流信号、外部转发链路、平台异常。这些事实跑完一周之后就拿不到了
- pending-notes 里如果指定了"周末复盘要点"，原样吸收进正式复盘文件
- 写完正式复盘并验证无误后，第七步完成时**一并 `rm` 掉 pending-notes 文件**（CLAUDE.md 里"流量观察工作流"已明确：复盘完成后删掉 pending-notes）
- 这一步漏掉过一次（2026-05-15 写 5/4-5/8 复盘时没读 pending-notes，错过了 5/6 X 限流这条关键事实，复盘里把账号势能问题误归为题材问题）。从那次起这一步硬性强制

## 第二步：抓取数据

- 用 Read 读 `README.md`，定位 Articles 表格
- 提取落入范围的文章的：`#`、日期、标题、文件路径、X 流量
- 至少保留 3 篇，否则提醒用户样本太小

## 第三步：读文章开头

对每篇文章：

- 用 Read 读文件前 30-40 行（开头 + 第一小节开头）
- 记录开头的**框架**：是现状切入 / 故事切入 / 身份呼唤 / 数字切入 / 报菜名
- 记录开头是否给出"这事跟你有关"的锚点

只读开头，不读全文。目的是判断**读者的 10 秒判断**，不是内容深度。

## 第四步：按 framework.md 打分

对每篇文章按 framework.md 的"高流量四支柱"打分（每条 0/1）：

- 读者身份一秒锁定
- 诊断 + 自救结构
- 具体数字 + 反差
- 情绪钩子明确

同时标注是否踩了"低流量失败模式"里的任何一条（标题悬疑、周末发、主角距离远、受众过窄、开头铺垫太长等）。

## 第五步：找模式，不要硬套

- 把文章按流量排序，高组 vs 低组做对比
- 找出**高组有、低组没有**的共性（正向规律）
- 找出**低组踩的、高组没踩的**坑（负向规律）
- 如果样本 ≥5 篇，观察星期几 vs 流量的关系，但明确标注样本不足时仅供参考
- 不要把单次偶然当规律：至少 2 篇印证才写进正向规律

## 第六步：生成建议

分两栏：

- **该多发**：受众基数大、有情绪钩子、有行动指南、反常识数字、平台元话题
- **该避免**：踩到的具体坑 + 本周观察到的失败模式

再给一条**单条最高杠杆动作**：这批数据里，如果只改一件事，最该改什么。必须落到具体可执行层面。

## 第七步：输出

按 `output-template.md` 的结构写入：

- 路径：`analysis/YYYY-MM-DD 至 MM-DD N篇流量复盘.md`（ISO 日期便于排序）
- 正文全中文，段落 2-4 句
- 禁破折号（——）
- 禁 AI 高频词（至关重要、意义重大、标志着、见证、根植于 等）
- 否定平行句式"不是X，是Y"最多 1 次
- 数字可读性：3.8k 保留原样，百分比保留原样，万数用"2000 万"而非 20,000,000

写完正式复盘 + 通过自检之后，立刻 `rm analysis/pending-notes-*.md` 把临时文件清掉。这是 CLAUDE.md "流量观察工作流"里的硬性约定，不要把临时观察留在仓库里。

写完后向用户返回：文件路径、样本文章数、最高流量/最低流量、一条最关键洞察。

## 第八步：下周计划（如果用户要求）

**重要约定：下周计划和流量预测写入同一份复盘文件，不要新开文件。**

如果用户在复盘之后要求"写下周计划"或"预测下周流量"，在当前复盘文件末尾追加：

1. 一级标题 `# 下周计划（M/D-M/D）` 作为分段
2. 子章节延续复盘的中文数字编号（复盘用到几，下周计划从几+1 开始）
3. 子章节固定包含：发布策略 + 日程表、逐篇预测（含方法说明）、风险点、复盘时机
4. 预测方法按 `.claude/skills/traffic-review/framework.md` 四支柱 + 历史 comp + 时间修正
5. 不写"硬性目标"；只给区间、中位数、观察点、下期复盘问题

同一份 md 文件同时包含复盘（过去）和计划（未来）。下一次复盘再新开文件，并在开头引用上一份计划做预测偏差对比。

## 规则追加

用户看完复盘后如果指出"这条规律其实不对"或"还有一个我一直没注意到的规律"，append 到 `framework.md` 对应章节，不修改 SKILL.md 或历史内容。
